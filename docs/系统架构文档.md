### 4.3 AI 对话（核心接口）

| 方法 | 路径 | 描述 |
|---|---|---|
| POST | `/conversations/:id/chat` | 发送消息，SSE 流式返回 AI 回复 |
| DELETE | `/conversations/:id/messages/:msgId` | 删除消息 |
| POST | `/conversations/:id/messages/:msgId/regenerate` | 重新生成 AI 回复 |

`/chat` 接口 SSE 响应格式（完整版）：
```
data: {"type":"thinking_start"}
data: {"type":"memory","count":3,"items":[{"summary":"用户喜欢下午茶"}]}
data: {"type":"delta","content":"喵"}
data: {"type":"delta","content":"～主人好！"}
data: {"type":"done","emotion":"happy","action":"喵喵 的尾巴开始左右摇摆","emoji_id":"uuid或null","usage":{"input_tokens":120,"output_tokens":45}}
data: {"type":"emoji","id":"uuid","url":"https://cdn.miao.example.com/emojis/happy_001.webp"}
data: [DONE]
```

字段说明：
- `emotion`：AI 自评情绪，对应颜文字映射表的 emotion_tag
- `action`：动作描述文字，可为 null；不为 null 时 App 显示动作提示条
- `emoji_id`：触发表情包的 ID，null 表示本次不发表情包；非 null 时服务器会紧随发一个 `type:emoji` 事件
- `type:emoji` 事件：App 在文字气泡之后延迟 600ms 插入表情包气泡

2. App 直接 PUT 到 `upload_url`（OSS 预签名，不过喵服务器，节省带宽）。
3. 上传完成后，`file_url` 作为消息内容的一部分发送给 `/chat` 接口。

```
用户发送消息
    │

    ▼
3. 请求 AI Provider（服务器代理）
    │
    ▼
4. SSE 流式回传 App
    │
    ▼
5. 保存消息到数据库
    │
    ▼
```

## 六、安全设计

| 项目 | 方案 |
|---|---|
| API Key 存储 | AES-256-GCM 加密，密钥通过环境变量注入，不落磁盘明文 |

## 七、微信接入架构（openclaw-weixin）

### 7.1 技术原理

微信接入基于腾讯开源的 **[openclaw-weixin](https://github.com/Tencent/openclaw-weixin)** 插件，配合 **Clawbot 扫码登录 API** 完成。整个流程分两个阶段：

**阶段一：扫码登录，获取 bot_token**
用户在 App 中扫描由 Clawbot API 下发的二维码，确认后获得 `bot_token`（Bearer Token）和对应的 `baseurl`。这个 token 是后续所有微信通信的唯一凭据，**无需 AppID、SecretKey 等任何预配置**。

**阶段二：持久消息通道（openclaw-weixin iLink Bot API）**
拿到 `bot_token` 后，喵服务器用它驱动 openclaw-weixin 的 iLink Bot API：
- 用 `getupdates` 长轮询持续接收微信消息
- 用 `sendmessage` 将 AI 回复发回微信

两个阶段完全解耦，`bot_token` 是桥梁。

### 7.3 Clawbot 扫码 API（获取 token）

**无需任何预配置**，直接调用：

| 步骤 | 接口 | 说明 |
|---|---|---|
| 1 | `POST /api/v1/wechat/qrcode` | 获取二维码，返回 `qrcode_url`（图片 URL）和 `qrcode`（轮询 token） |
| 2 | `POST /api/v1/wechat/qrcode/status` | 长轮询，传入 `qrcode` token，状态：`wait / scaned / confirmed / expired` |
| 3 | `status=confirmed` | 响应中 `credentials.bot_token`、`credentials.baseurl` 即为最终凭据 |
| 解绑 | `POST /api/v1/wechat/channel_reset` | 传入 `channel_id`，停止长轮询并清除通道记录 |

`confirmed` 时响应结构：
```json
{
  "data": {
    "status": "confirmed",
    "credentials": {
      "bot_token": "xxxxx",
      "ilink_bot_id": "xxxxx",
      "ilink_user_id": "xxxxx",
      "baseurl": "https://xxx.weixin.qq.com"
    }
  }
}
```

### 7.4 iLink Bot API（收发消息）

openclaw-weixin 的所有接口均为 `POST JSON`，base URL 来自扫码时获得的 `baseurl`，**无需 AppID/SecretKey**，仅需 Bearer Token：

```
Authorization:     Bearer <bot_token>
AuthorizationType: ilink_bot_token
X-WECHAT-UIN:      base64(decimal(random_uint32))
Content-Type:      application/json
```

每个请求体均附带 `base_info`：
```json
{ "channel_version": "2.0.0", "bot_agent": "Miao/1.0" }
```

**核心接口：**

| 接口 | Path | 说明 |
|---|---|---|
| 长轮询收消息 | `POST {baseurl}/ilink/bot/getupdates` | 传 `get_updates_buf` 游标，服务端持有请求直到有消息（≤35s），返回 `msgs[]` + 新游标 |
| 发送消息 | `POST {baseurl}/ilink/bot/sendmessage` | 传 `msg.to_user_id`、`msg.context_token`（必须从收到的消息中echo回去）、`msg.item_list` |
| 获取配置 | `POST {baseurl}/ilink/bot/getconfig` | 获取 `typing_ticket`（用于发送输入状态） |
| 发送输入状态 | `POST {baseurl}/ilink/bot/sendtyping` | `status: 1=typing / 2=cancel` |
| 通知启动 | `POST {baseurl}/ilink/bot/msg/notifystart` | Worker 启动时调用，告知服务端开始推送 |
| 通知停止 | `POST {baseurl}/ilink/bot/msg/notifystop` | Worker 停止/解绑时调用 |

**消息结构（WeixinMessage）核心字段：**

```typescript
{
  from_user_id:   string,   // 发送方微信ID
  to_user_id:     string,   // 接收方（bot）ID
  session_id:     string,   // 会话标识
  context_token:  string,   // 回复时必须echo此值
  message_state:  0|1|2,    // 0=NEW 1=GENERATING 2=FINISH
  message_type:   1|2,      // 1=USER 2=BOT
  item_list: [{
    type: 1|2|3|4|5,        // 1=TEXT 2=IMAGE 3=VOICE 4=FILE 5=VIDEO
    // type=1 时:
    text: { text: string }
  }]
}
```

只处理 `message_type=USER`（用户发来的消息）且 `message_state=FINISH` 的消息，忽略 BOT 自身回显。

### 7.5 服务器侧 WechatBotWorker

每个绑定的微信账号对应一个独立的 Worker 实例（Node.js async loop）：

```
WechatBotWorker
├── 启动：调用 notifystart，初始化 cursor=''
├── 主循环：getupdates → 处理消息 → sendTyping → AI生成 → sendMessage → 更新cursor
├── 错误处理：网络超时(35s)视为正常，重试；连续失败5次标记binding为error状态
└── 停止：调用 notifystop，清理资源
```

Worker 生命周期由后台管理端控制：
- 管理员在后台可查看所有 binding 的 Worker 运行状态（运行中 / 已停止 / 错误）
- 支持手动重启单个 Worker
- 服务器重启时自动重建所有 `is_active=true` 的 Worker

### 7.6 数据库

```sql
-- 微信绑定表
CREATE TABLE wechat_bindings (
  id             UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id        UUID REFERENCES users(id) ON DELETE CASCADE,
  bot_token      TEXT NOT NULL,          -- AES-256-GCM 加密存储
  ilink_bot_id   VARCHAR(128),
  ilink_user_id  VARCHAR(128),
  base_url       VARCHAR(512) NOT NULL,  -- 来自 credentials.baseurl
  worker_status  VARCHAR(16) DEFAULT 'stopped', -- running/stopped/error
  is_active      BOOLEAN DEFAULT true,
  last_error     TEXT,                   -- 最近一次错误信息
  created_at     TIMESTAMPTZ DEFAULT now(),
  updated_at     TIMESTAMPTZ DEFAULT now(),
  UNIQUE(user_id)                        -- 每个用户只允许绑定一个微信
);

-- messages 表新增来源字段
ALTER TABLE messages ADD COLUMN source VARCHAR(16) DEFAULT 'app';
-- source: 'app' | 'wechat'
```

### 7.7 后台管理端负责内容

后台管理端**不需要配置任何微信凭据**，仅负责：

| 功能 | 说明 |
|---|---|
| 查看绑定列表 | 所有用户的微信绑定状态（已绑定 / 未绑定 / 错误） |
| Worker 状态监控 | 每个绑定账号的 Worker 是否正常运行、最近活跃时间 |
| 手动重启 Worker | 某个账号出现异常时，管理员可手动重启 |
| 强制解绑 | 管理员可强制解除某用户的微信绑定（调用 channel_reset + 停止 Worker） |
| 消息统计 | 各账号的微信消息量统计 |
